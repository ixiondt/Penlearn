## You can't test what you haven't found

A web app's real attack surface is rarely the links in the nav bar. It's the
`/admin` panel nobody linked, the `/api/v1/` endpoints, the `.git/` directory
that shipped by accident, the GraphQL endpoint with introspection left on.
Content discovery and fuzzing find that hidden surface so the rest of the
assessment has something to work on.

`web-fuzzer.sh` and `webapp-scanner.sh` cover discovery and rapid assessment.

## Content discovery

```bash
./scripts/web-fuzzer.sh https://example.com --api-discovery --engagement client-2026
```

Under the hood it drives the standard discovery tools and adds targeted checks:

- **gobuster / ffuf** — directory and file brute-forcing against wordlists
- **nuclei** — templated checks for known exposures and misconfigurations
- **SSRF parameter probing** — tries ~22 common parameter names against a cloud
  metadata endpoint (the classic `169.254.169.254` test)
- **GraphQL introspection** — discovers GraphQL endpoints and dumps the schema if
  introspection is enabled (it usually shouldn't be in prod)
- **File-upload extension testing** — throws 21 dangerous extensions at discovered
  upload endpoints to find weak server-side validation

## Rapid assessment

`webapp-scanner.sh` is the quick triage pass:

```bash
./scripts/webapp-scanner.sh https://example.com --engagement client-2026          # quick
./scripts/webapp-scanner.sh https://example.com --full --engagement client-2026   # + rate-limit, API versions
```

Quick mode checks security headers, CORS, cookie flags, info disclosure, SQLi/
XSS/SSTI *indicators*, auth-endpoint discovery, and open-redirect tests. Full mode
adds rate-limiting assessment and API version enumeration. Output is a
severity-tagged finding list.

## Discovery feeds everything else

```text
web-fuzzer.sh / webapp-scanner.sh
        │ discover endpoints, params, uploads, GraphQL
        ▼
injection / auth / WAF-bypass testing on what was found
        │
        ▼
findings.json ─▶ risk-scoring ─▶ report
```

Discovery isn't the assessment — it's what makes the assessment possible. An
injection test needs a parameter; an IDOR test needs an endpoint; an upload
exploit needs an upload form. Fuzzing finds the inputs the rest of the web track
attacks.
