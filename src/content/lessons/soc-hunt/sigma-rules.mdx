## What Sigma is, and isn't

Sigma is a YAML format for describing SIEM detection rules in a vendor-neutral way. You write
the rule once; converters translate it to Splunk SPL, KQL (Sentinel/Defender), Elastic DSL,
Sumo Logic, QRadar AQL, etc. The format is deliberately simple — `detection`, `condition`,
`falsepositives` — and that simplicity is the value.

It's not:

- A SIEM. You still need somewhere to run the rule.
- A complete language. Some logics (graph traversal, multi-event correlation across long windows) don't translate cleanly.
- A guarantee of low false-positive rate. That's still your job.

The toolkit's `scripts/sigma-rule-builder.sh` generates Sigma rules from ATT&CK technique IDs.
26 IT + 6 ICS techniques are pre-templated, including the CCTC anti-forensics/privesc set
and the AiTM phishing set (T1539, T1078.004, T1556.006).

## A minimal Sigma rule

```yaml
title: Suspicious PowerShell EncodedCommand
id: 4f3d20a1-1c11-4d3a-b2c5-6e21f8b59a14
status: experimental
description: Detects PowerShell launched with -EncodedCommand from a non-administrator session.
references:
  - https://attack.mitre.org/techniques/T1059/001/
author: penlearn
date: 2026-03-05
tags:
  - attack.execution
  - attack.t1059.001
logsource:
  product: windows
  service: process_creation
detection:
  selection:
    Image|endswith: '\powershell.exe'
    CommandLine|contains:
      - '-EncodedCommand'
      - '-enc '
      - '-e '
  filter:
    User|startswith: 'NT AUTHORITY\\'
  condition: selection and not filter
falsepositives:
  - Legitimate admin scripts using EncodedCommand for quoting reasons
level: medium
```

Read it as: *match events where Image is powershell.exe AND CommandLine contains an encoded-command
flag, EXCEPT when the user is a NT AUTHORITY system account.*

Three things make this a usable rule rather than a noisy one:

1. **A filter for known-noise.** Without `not filter`, you'd see thousands of legit system-context events.
2. **An explicit log source.** Without `logsource`, the converter doesn't know what data to query.
3. **Documented false positives.** Tells the next analyst what to look for when triaging hits.

## Generating from a technique

```bash
./scripts/sigma-rule-builder.sh --technique T1059.001 --incident-id INC-2026-001
```

Produces rules like the above, parameterized for your environment. The script prints tunables
the operator must fill in:

```text
NOTE: replace <your-domain> regex with the actual AD domain
NOTE: replace <vpn-egress-asn> allowlist with your actual VPN egress ASNs
NOTE: confirm svc-/sa-/-bot service-account naming convention matches your environment
```

These notes are not optional. A rule deployed without filling them in either fires constantly
(no filter populated) or never fires (over-filtered placeholder values).

## Tuning before deployment

A generated rule is a starting point. Two-step tuning:

### Step 1: Backtest

Run the rule against historical telemetry — typically 30-90 days. Count the hits. If you
get more than a handful per day, the rule will be ignored in production. Tighten the
filter until you have one to five hits/day in a real environment.

### Step 2: Hit triage on samples

For each historical hit, classify as TP / FP / unknown. If your FP rate is >20%, the rule
needs a better filter. The Sigma `falsepositives` field should list every common FP class
you found.

The toolkit's generated rules ship with reasonable defaults, but defaults can't know your
environment. Service account names, VPN egress, admin tool baselines — those are yours
to fill in.

## Where rules live

The toolkit's convention:

```text
~/incidents/[INCIDENT-ID]/08-aar/detection-rules/
~/hunts/[HUNT-ID]/04-detection-rules/
~/restoration/[INCIDENT-ID]/04-detection-rules/
```

For organization-level deployment, you usually have a separate detection-engineering repo.
The Sigma rule travels from per-incident → org repo → SIEM via your normal CI pipeline.

## Integrating with the SecOps toolkit pipeline

Rules generated during an IR engagement get bundled into:

- The AAR (after-action report) so the lessons are durable
- The VECTR coverage push (`scripts/vectr-sync.sh`) — so coverage scoring updates
- The CTI export (`scripts/cti-export.sh`) — IOCs alongside the rule that catches them

The integration matters because rules that aren't tracked become orphans. VECTR is the
toolkit's preferred coverage tracker because it ties detection rules to ATT&CK techniques
to threat scenarios.

## Action items

1. Generate a Sigma rule for one technique you care about: `./scripts/sigma-rule-builder.sh --technique T1059.001`.
2. Read every `NOTE:` and fill in the tunables.
3. Backtest against your SIEM's historical data.
4. Iterate the filter until daily hit count is manageable.
5. Document the FP classes you found in the `falsepositives:` field.
6. Push to VECTR via `./scripts/vectr-sync.sh` so coverage tracking knows about it.
